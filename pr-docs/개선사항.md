# 프로젝트 분석 및 개선사항

> 분석 기준: 0-mvp phase 완료 시점 (2026-04-25)
> 대상: backend (Spring Boot) + web (Next.js)

---

## 1. 백엔드

### Critical

#### 1-1. Testcontainers IT 테스트 전체 skip
- **위치**: `MatchServiceConcurrencyIT`, `ChatFlowIT`, `HelpRequestRepositoryIT`, `MatchDispatcherIT`
- **원인**: Docker Desktop 4.69 소켓 호환 이슈. `disabledWithoutDocker=true`로 건너뜀.
- **영향**: 동시성 확정 경쟁(가장 중요한 비즈니스 규칙)이 실제로 한 번도 검증되지 않음.
- **개선**: `TESTCONTAINERS_DOCKER_SOCKET_OVERRIDE` 환경변수 설정 또는 Docker Desktop > Settings > General > "Use Docker socket" 활성화. CI 환경에서는 `testcontainers.reuse.enable=true` + Ryuk 비활성화 조합.

#### 1-2. 가이드 온듀티 TTL 없음
- **위치**: `RedisGeoService.addGuide()` — TTL 미설정
- **영향**: 앱/서버 비정상 종료 시 가이드가 Redis에 영구 on-duty 상태로 남아 오프라인 가이드에게 요청 알림이 계속 발송됨.
- **개선**: `addGuide()` 시 `guide:location:{guideId}` heartbeat 키를 24시간 TTL로 함께 저장. ScheduledTask가 주기적으로 heartbeat 키가 없는 guideId를 GEO에서 제거.

#### 1-3. MatchService → ChatService 직접 의존 (도메인 결합)
- **위치**: `MatchService.java:50` — `ChatService` 생성자 주입
- **영향**: `match` 도메인이 `chat` 도메인에 직접 의존. 두 도메인을 독립적으로 테스트하거나 배포하기 어려움. ARCHITECTURE.md의 "도메인 간 참조는 서비스 인터페이스로만" 규칙 위반.
- **개선**: `MatchConfirmedEvent` 도메인 이벤트를 발행하고 `chat` 도메인에서 `@EventListener`로 채팅방 생성. 혹은 `ChatRoomCreationPort` 인터페이스를 두고 `chat` 모듈이 구현.

#### 1-4. 결제 의도 중복 생성
- **위치**: `PaymentService.createPaymentIntent()`
- **영향**: 같은 `requestId`로 여러 번 호출 시 `PaymentIntent` 레코드가 중복 생성됨.
- **개선**: 서비스 진입 시 `requestId`로 기존 `PaymentIntent` 조회. 존재하면 기존 레코드 반환(멱등). 없을 때만 생성. `payment_intent` 테이블에 `request_id UNIQUE` 제약 추가(V8 마이그레이션).

### Important

#### 1-5. @WebMvcTest 컨트롤러 테스트 미구현
- **위치**: `UserControllerTest.java`만 존재. `MatchController`, `PaymentController`, `ReviewController`, `ChatController`는 테스트 없음.
- **영향**: 잘못된 HTTP 상태코드, 응답 직렬화 오류, 권한 누락을 빌드 시점에 잡지 못함.
- **개선**: 각 컨트롤러별 `@WebMvcTest` 클래스 작성. 최소 200/401/403/422 경로 커버.

#### 1-6. Rate Limiting 없음
- **위치**: `SecurityConfig.java` — `/auth/signup`, `/auth/login` 무제한
- **영향**: 브루트포스 / 크리덴셜 스터핑 공격 무방비.
- **개선**: Spring Boot의 Bucket4j 라이브러리 또는 Redis 기반 카운터를 `JwtAuthenticationFilter` 앞에 추가. IP당 분당 20회 제한이 기본값.

#### 1-7. Swagger/OpenAPI 미구현
- **위치**: `SecurityConfig.java:31`에 `/swagger-ui/**`, `/v3/api-docs/**` 경로는 열려있지만 `springdoc-openapi-starter-webmvc-ui` 의존성과 설정 없음.
- **개선**: `build.gradle`에 `springdoc-openapi-starter-webmvc-ui:2.x` 추가. `@Tag`, `@Operation` 주요 엔드포인트에 적용. 모바일 앱 개발 시 API 계약 확인 도구로 활용.

#### 1-8. JWT Access Token 단독 운용
- **영향**: 토큰 만료 시 재로그인 필요. 모바일 앱에서 특히 UX가 나쁨.
- **개선**: Refresh Token 발급 + `POST /auth/refresh` 엔드포인트 추가. Access Token 15분, Refresh Token 7일(Redis 블랙리스트 관리).

#### 1-9. CORS 설정 명시 없음
- **위치**: `SecurityConfig.java` — `.cors()` 설정 없음. Spring Boot 기본값 적용 중.
- **영향**: 모바일 앱은 CORS 무관하지만, 웹 클라이언트가 다른 도메인에서 직접 백엔드를 호출하는 경우 차단됨. 명시적 출처 허용 필요.
- **개선**: `CorsConfigurationSource` Bean을 등록하고 허용 출처를 환경변수(`ALLOWED_ORIGINS`)로 주입.

#### 1-10. Outbox 패턴 미구현 (ADR-003 TODO)
- **위치**: `RabbitPublisher` — AFTER_COMMIT 직후 발행 실패 시 이벤트 유실
- **영향**: 결제 완료 후 정산 이벤트 유실 가능.
- **개선**: `outbox` 테이블 추가, 커밋과 동일 트랜잭션에서 레코드 삽입, 별도 `@Scheduled` 폴러가 발행 후 삭제. ADR-003 재검토.

### Minor

#### 1-11. `review` 도메인 테스트 누락
- `ReviewServiceTest.java`는 존재하지만 `ReviewController` 테스트 없음. 리뷰 제출 후 가이드 평점 집계(`avgRating`, `ratingCount`) 업데이트 경로를 통합 테스트로 검증 필요.

#### 1-12. API 버전 prefix 없음
- 모든 엔드포인트가 `/auth/`, `/requests/` 등 루트에 직접 노출. 모바일 앱 출시 후 하위 호환성 관리가 어려움. `/v1/` prefix 도입 권장 (ADR로 결정).

## 2. 웹 클라이언트 (Next.js)

### Critical

#### 2-1. Playwright e2e 테스트 미구현
- `package.json`에 `"e2e": "playwright test"` 스크립트가 있으나 `playwright.config.ts`가 없고 테스트 파일도 없음.
- ARCHITECTURE.md에 명시된 핵심 시나리오("로그인 → 여행자 요청 생성 → 가이드 수락 → 확정 → 채팅 1회")가 검증되지 않음.
- **개선**: `playwright.config.ts` 생성 및 핵심 플로우 1개 e2e 테스트 구현.

#### 2-2. JWT 만료 처리 없음
- Route Handler가 401 응답을 받아도 쿠키 삭제 + `/login` 리다이렉트를 하지 않음.
- **개선**: `api-client.ts`에서 401 응답 시 `clearAuthToken()` 호출 후 `NextResponse.redirect('/login')` 처리.

### Important

#### 2-3. 가이드 뷰 3초 폴링
- **위치**: `GuideView.tsx` 또는 `RequestCard.tsx` — `refetchInterval: 3000`
- **영향**: 불필요한 HTTP 요청 과다. STOMP `NEW_REQUEST` 이벤트를 이미 `RealtimeProvider`가 수신하는데 활용하지 않음.
- **개선**: `RealtimeProvider`의 `NEW_REQUEST` 이벤트에서 `queryClient.invalidateQueries(['nearbyRequests'])` 호출 후 폴링 제거.

#### 2-4. Error Boundary 미적용
- TravelerView, GuideView, ChatPanel 모두 에러 시 전체 페이지 크래시.
- **개선**: `app/traveler/page.tsx`, `app/guide/page.tsx`에 React Error Boundary 래핑. fallback UI에 "새로고침" 버튼.

#### 2-5. 현재 위치 폴백 없음
- `RequestForm.tsx`에서 `navigator.geolocation.getCurrentPosition` 거부 시 폼이 전송 불가 상태로 멈출 가능성.
- **개선**: 위치 권한 거부 시 수동 좌표 입력 필드 표시 + 기본값(서울 시청 좌표) 제안.

#### 2-6. STOMP 연결 실패 재시도 누락
- **위치**: `stomp-client.ts` — 연결 실패 시 재시도 로직 없음.
- **개선**: `reconnectDelay: 5000` 설정. `onDisconnect`에서 연결 상태를 상위 컴포넌트(RealtimeProvider)로 전파해 상태 인디케이터에 반영.

### Minor

#### 2-7. 컴포넌트 단위 테스트 커버리지 부족
- `ChatPanel.test.tsx`, `GuideOfferCard.test.tsx`, `OnDutyToggle.test.tsx`, `RequestCard.test.tsx` 4개 존재. `TravelerView`, `GuideView`, `LoginForm`, `ReviewForm`은 미구현.

## 3. 아키텍처 / 공통

### 3-1. 도메인 간 결합도 정리
- `match → chat` 직접 의존 (1-3 참조).
- `payment → request` 직접 의존도 확인 필요. `PaymentService`에서 `HelpRequest` 상태를 직접 변경한다면 동일 문제.

### 3-2. 관측성 보강
- 현재 `requestId`만 MDC에 기록. `userId`, `requestType`, `matchId`를 추가하면 로그 추적 효율 상승.
- 결제 상태 전이, 매칭 확정 같은 중요 이벤트에 구조화 로그(`log.info("match.confirmed", Map.of("requestId", ..., "guideId", ...))`) 추가.

### 3-3. 프로파일 분리 미흡
- `application-local.yml`만 gitignore. `application-test.yml`, `application-staging.yml` 프로파일 없음. CI에서 실제 DB 대신 H2/Testcontainers를 명시적으로 선택하는 구조 필요.

## 4. 모바일 앱 전환 시 추가 고려사항 (신규 phase)

아래 항목들은 `1-mobile-app` phase 설계 시 사전에 결정해야 할 사항이다.

| 항목 | 웹(현재) | 모바일(신규) |
|------|---------|------------|
| JWT 저장 | HttpOnly 쿠키 (Next.js Route Handler) | `expo-secure-store` |
| API 호출 경로 | BFF 프록시(`/api/**`) | 백엔드 직접 호출 |
| WebSocket | SockJS + STOMP | Native WebSocket + STOMP.js |
| 지도 | Leaflet + OSM | `react-native-maps` + OSM |
| 위치 정보 | `navigator.geolocation` | `expo-location` (권한 요청 필요) |
| CORS | Next.js 서버가 처리 | 백엔드에 CORS 설정 필수 |
| Push 알림 | 없음 (STOMP 구독만) | Expo Notifications (FCM/APNs) |
