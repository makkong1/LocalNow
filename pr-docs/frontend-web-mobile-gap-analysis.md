# 웹·모바일 프로트 갭 분석 (기능·플로우·디자인)

**작성 기준:** 저장소 실제 코드(`web/`, `mobile/`), `docs/PRD.md`, `docs/UI_GUIDE.md`, `CLAUDE.md` (프론트 규약).  
**목적:** 데모 웹과 진행 중 모바일 앱의 **현재 구현 대비 부족한 점**을 PR·로드맵 논의용으로 정리한다.

---

## 요약

| 구분        | 웹 (`web/`)                                                                                           | 모바일 (`mobile/`)                                               |
| ----------- | ----------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| 역할        | Next.js 데모 클라이언트 + Route Handler BFF, **데스크톱 시연** 고정                                   | Expo RN 앱, **백엔드 직접 호출**, 필드 사용 전제                 |
| 강점        | 역할별 라우트 분리(`/traveler`·`/guide`), 어드민 읽기 전용 대시보드, UI 가이드와 Tailwind 정합성 높음 | 실제 GPS·지도·탭 내비, 매칭 후 결제·리뷰 화면까지 스택 분리      |
| 공통 취약점 | 요청 **취소·프로필 수정·푸시** 등 운영 기능 부재, PRD상 **i18n·접근성** 미포함                        | 동일 + **채팅 탭 플레이스홀더**, OAuth 브라우저 플로우 설정 의존 |

## 1. 기능 갭

### 1.1 웹 전용

- **관리자 집계(`/admin`)**  
  `UserRole.ADMIN` 전용 읽기 대시보드는 **웹에만** 있다 (`docs/PRD.md`, ADR-014와 일치). 모바일에는 동등 화면 없음 — 의도된 범위면 문서·온보딩에 “운영은 웹”만 명시하면 된다.

- **위치 고정(서울)**  
  `TravelerView`는 지도·요청 좌표가 **서울 고정**(`userLocation = SEOUL`). 실제 GPS 없이 두 탭 시연용으로는 타당하나, “실제 여행 시나리오”와는 거리가 있다.

### 1.2 모바일 전용

- **하단 ‘채팅’ 탭**  
  구현은 플레이스홀더(“매칭 확정 후 채팅이 열립니다”)뿐이고, 실제 채팅은 **스택 `ChatRoom`**으로만 진입한다. 탭 네이밍과 내용이 불일치해 **첫 사용자에게 혼란**을 준다.

- **역할과 UI**  
  웹은 `/traveler`·`/guide`에서 **역할 불일치 시 리다이렉트**한다. 모바일은 **Traveler / Guide 탭을 역할과 무관하게 항상 노출**한다. 백엔드가 거부할 수 있어도, 제품 관점에서는 가이드 전용·여행자 전용 탭 숨김 또는 진입 시 안내가 없다.

### 1.3 공통 (웹·모바일 모두)

- **요청 취소(UI)**  
  상태 뱃지에 `CANCELLED` 표시는 있으나, 사용자가 진행 중 요청을 **취소하는 버튼·플로우**는 프론트에서 확인되지 않는다 (백엔드 API 존재 여부와 별개로 **클라이언트 공백**).

- **프로필·설정**  
  회원가입 시 역할·도시 등 일부 필드는 있으나, 로그인 후 **프로필 수정**(언어·도시·표시 이름 등) 화면은 양쪽 모두 없다.

- **완료 후 재요청·히스토리**  
  웹 `TravelerView`는 여러 요청 카드 선택 가능. 모바일은 `getActiveRequest`로 단일 “활성” 요청 위주 — **복수 요청·과거 목록** UX가 웹과 완전히 같지 않다.

- **결제·리뷰**  
  웹: 우측 패널에서 Mock 캡처·`ReviewForm`. 모바일: 전용 `Payment`·`Review` 스크린. **기능은 대응**하나 위치·진입 경로가 달라 **교육·테스트 시나리오를 따로 가져가야** 한다.

- **알림**  
  매칭·메시지 **푸시·배지** 없음 (PRD MVP 범위 밖일 수 있음 — 다만 “실제 앱” 관점 갭으로 남는다).

- **국제화·접근성**  
  PRD상 한/영 하드코딩·반응형·접근성 튜닝 제외. 디자인·플로우 평가 시 **의도적 제외**로 표시하는 것이 맞다.

## 2. 플로우·정합성 갭

### 2.1 인증

| 항목          | 웹                            | 모바일                                   |
| ------------- | ----------------------------- | ---------------------------------------- |
| 이메일 로그인 | BFF + 쿠키 기반               | `api-client` + Secure Store              |
| OAuth         | Next 라우트·`/oauth/callback` | 시스템 브라우저로 백엔드 OAuth2 URL 오픈 |

모바일 `oauth-redirect.ts` 주석대로, 완료 URL은 **기본적으로 Next 콜백 호스트**를 가정한다. 실기기·Expo에서는 **LAN IP·앱 링크 스킴** 등 추가 설정 없이는 OAuth 완주가 깨지기 쉽다 — **개발자 경험·온보딩 문서화 갭**.

### 2.2 실시간

- 웹: `RealtimeProvider` + 토스트 등으로 매칭 이벤트 피드백.
- 모바일: 가이드는 STOMP 구독으로 `MATCH_CONFIRMED` 처리 등 **역할별 구현 차이**.  
  양쪽 모두 **연결 상태**는 헤더/일부 UI에 있으나, **재연결 중·끊김**에 대한 UI 가이드(`docs/UI_GUIDE.md`의 인디케이터 규칙) 적용 수준은 컴포넌트마다 다를 수 있다.

### 2.3 API 계약

- `mobile/src/types/api.ts`·`web/src/types/api.ts` 모두 `docs/API_CONVENTIONS.md`와 맞추도록 되어 있으나, **두 클라이언트 타입 파일은 별도 유지**다. 필드 추가 시 **한쪽만 업데이트**하면 불일치 위험이 있다 — 프로세스·CI 점검 갭.

## 3. 디자인·UX 갭

### 3.1 `docs/UI_GUIDE.md` 관점

- 문서는 명시적으로 **웹 단일 기준**이다. 웹은 Tailwind 토큰(배경 `#141414`, amber 포인트 등)과 **대체로 일치**한다.
- 모바일은 `StyleSheet`/컴포넌트에 동일 palette를 쓰지만, **NativeWind 전역 토큰 vs 수동 상수** 혼용 가능성이 있어 장기적으로 **웹과 미세한 간격·타이포 차이**가 누적될 수 있다.
- PRD·레이아웃: 웹은 `max-w-6xl`·12컬럼 그리드. 모바일은 단일 컬럼·탭 — **문서의 “섹션 라벨 uppercase” 등**은 웹에 더 가깝게 맞춰져 있다.

### 3.2 웹

- **뷰포트:** `layout.tsx`에서 `min-w-[1024px]` 및 작은 화면 안내 배너 — PRD “데스크톱 1280 기준”과 정합. **모바일 브라우저 사용자에게는 의도적으로 불친절** (제품 선택).

- **랜딩:** `/`는 `/login` 리다이렉트만 — 마케팅 페이지 없음. PRD “도구형”과 일치.

### 3.3 모바일

- **채팅 탭:** 위 기능 갭과 동일 — 네비게이션 **정보 구조(IA) 수정** 또는 채팅 목록 화면 도입이 필요해 보인다.

- **역할 미분리 UI:** 한 계정으로 데모할 때는 유용하지만, 실사용 시 **불필요 탭 노출**은 디자인·제품 정책과 충돌할 수 있다.

## 4. 심각도 정리 (제안)

**판정 기준 (요약)**

| 등급 | 의미 |
|------|------|
| **차단** | 일반 사용자·실사용 시나리오에서 **막히거나 오해를 유발**하여 출시 전에 결정·수정이 필요한 항목 |
| **권장** | 당장 크래시는 아니나 **규약·운영·유지보수** 관점에서 다음 마일스톤에 넣기 적합 |
| **선택** | PRD·MVP 범위 밖이거나 **품질·확장** 단계에서 다룰 과제 |

### 차단 (제품 출시 전 검토)

- **모바일 OAuth** — 성공 리다이렉트 호스트·실기기 네트워크와 맞지 않으면 소셜 로그인 전 과정이 실패한다 (§2.1).
- **채팅 탭 IA** — 하단 탭 이름과 실제 진입 경로가 달라 **첫 실행 사용자에게 치명적 혼란**이다 (§1.2, §3.3). 탭 제거·이름 변경·매칭 후 활성화·목록 화면 중 하나로 정리 필요.

### 권장

- **역할 가드** — 웹과 동일하게 역할별 탭/화면 노출 또는 리다이렉트 (§1.2).
- **요청 취소·프로필 수정 UI** — 백엔드와 계약만 있다면 클라이언트 플로우 추가 (§1.3).
- **API 타입 동기화** — `web`/`mobile` `types/api.ts` 변경 시 체크리스트 또는 스크립트·CI 힌트 (§2.3).
- **실시간 연결 상태 UI** — `UI_GUIDE`의 connected / reconnecting / disconnected 규칙을 웹·모바일 주요 화면에 통일 (§2.2).

### 선택

- **푸시·인앱 배지** — 매칭·채팅 알림 (§1.3).
- **복수 요청·히스토리 UX** — 웹과 모바일 행동 맞추기 (§1.3).
- **접근성·반응형** — PRD에서 의도적으로 제외된 영역; 문서·ADR 범위를 넓힐 때 재평가 (§1.3, §3.2).
- **웹 위치 서울 고정** — 데모 유지 vs 실GPS 옵션 (§1.1).
